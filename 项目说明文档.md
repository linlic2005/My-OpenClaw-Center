# OpenClaw Center 项目说明文档

## 1. 项目定位

OpenClaw Center 是一个基于 React + TypeScript + Tauri 2 的 OpenClaw 客户端项目。

当前版本的目标不是单纯做一个传统后台，而是把三个方向整合到同一个产品里：

1. 用 `ClawX` 风格重做主控台界面
2. 用 `Star-Office-UI` 风格提供 Agent 工作室场景
3. 保留并强化 OpenClaw Gateway 的真实连接能力

现在的产品结构可以理解为：

```text
ClawX 风格主控台
  + OpenClaw Gateway 真连接
  + Star Office 风格工作室场景
  + Chat / Kanban / Configs / Studio / Settings 模块
```

## 3. 本次 Gateway 测试结果

本次我直接对当前默认本地 Gateway 地址做了端口探测：

- 地址：`127.0.0.1:18789`
- 时间：`2026-04-03`
- 命令：`Test-NetConnection 127.0.0.1 -Port 18789`
- 结果：`TcpTestSucceeded = False`

结论：

- 当前本机 `127.0.0.1:18789` 没有服务监听
- 所以项目“此刻”不能正常连接本地 Gateway
- 问题不在前端页面逻辑，而是在本机 Gateway 进程未启动、端口未暴露，或监听地址/端口不一致

这也意味着：

- 现在不能完成真实 WebSocket 握手验证
- 只有等本机 Gateway 正常监听后，才能继续验证 `connect.challenge -> connect -> agents.list`

建议下一步排查顺序：

1. 确认 OpenClaw Gateway 进程是否已启动
2. 确认它监听的是否真的是 `127.0.0.1:18789`
3. 若端口变更，更新 `.env` 或设置页里的 `Gateway URL`
4. 端口恢复后，再重新测试前端连通性

## 7. 关键服务层

### `GatewayService.ts`

这是最核心的协议层。

职责：

- WebSocket 连接管理
- `connect.challenge` / `connect` 握手
- 心跳
- 自动重连
- 离线队列
- 请求分发
- `agents.list` 拉取

如果要改 Gateway 协议，优先看这里。

### `GatewayDeviceAuth.ts`

职责：

- 生成或读取本地设备身份
- 生成签名
- 构造设备认证负载
- 存储配对后的 device token

### `PersistenceService.ts`

职责：

- 本地持久化统一入口
- 设置、错误日志、离线队列、设备身份等数据存取

后续如果增加本地缓存，不要在各个 service 里随便直接写 `localStorage`，优先走这个服务。

## 8. 状态层关系

项目使用 Zustand。

每个主要模块都有自己的 store：

- `gatewayStore.ts`
- `chatStore.ts`
- `kanbanStore.ts`
- `fileStore.ts`
- `studioStore.ts`
- `settingsStore.ts`

可以把它们理解成：

```text
React 组件
  -> store
  -> service
  -> Gateway / Studio / Persistence
```

维护原则：

- UI 改动优先改 `components`
- 页面状态优先改 `stores`
- 协议和持久化优先改 `services`

## 12. 已知维护风险

1. 旧模块内仍可能有残留乱码文案
   现在主壳层、Studio 场景和设置页已重写，但 Chat / Kanban / File 模块仍建议继续做一轮中文清理。

2. Gateway 实连依赖外部服务状态
   当前前端协议链路已经具备，但如果本机 Gateway 没启动，页面一定会表现为断开。

3. `dist/` 目录会随着构建变化
   若准备提交仓库，建议先确认是否要保留构建产物。
