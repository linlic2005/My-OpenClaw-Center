# OpenClaw Center 项目说明文档

## 1. 项目概述

OpenClaw Center 是一个基于 React + TypeScript + Tauri 2 的 OpenClaw 客户端项目。
它当前同时支持两类使用方式：

1. 本地模式
   Gateway 与 Studio 和客户端运行在同一台机器上。
2. 远程模式
   客户端运行在 Windows 或浏览器端，Gateway 与 Studio 部署在 Linux 服务器上。

项目的核心目标是为 OpenClaw 提供统一的聊天、看板、文件查看、工作区和设置管理界面，并通过官方 Gateway 协议完成真实连接，而不是使用本地 mock。

---

## 2. 技术栈

- 前端框架：React 18
- 语言：TypeScript
- 构建工具：Vite
- 状态管理：Zustand
- 桌面壳：Tauri 2
- 本地持久化：Tauri SQL / localStorage 回退
- 测试：Vitest + jsdom
- Studio 子服务：Flask

## 4. 前端架构说明

项目整体采用三层结构：

1. UI 组件层
   位于 `src/components/`，负责页面展示、交互和调用 store。
2. 状态层
   位于 `src/stores/`，负责聚合页面状态、调用 service、向组件提供响应式数据。
3. 服务层
   位于 `src/services/`，负责协议实现、数据处理、持久化和与后端服务通信。

可以理解为：

```text
React 组件 -> Zustand Store -> Service -> Gateway / Studio / Persistence
```

这种分层方式的维护好处是：

- UI 改动不会直接污染协议逻辑
- Gateway 协议变更可以集中在 service 层处理
- 状态更新和副作用较容易定位

## 5. 核心模块架构与职责

### 5.1 应用入口

文件：

- `src/main.tsx`
- `src/App.tsx`

职责：

- 挂载 React 应用
- 初始化主题、语言、字号、紧凑模式
- 启动 Gateway 连接
- 根据当前标签页切换模块内容

### 5.2 布局层

文件：

- `src/components/layout/Shell.tsx`

职责：

- 提供统一外壳
- 管理侧边栏、顶部状态、模块切换
- 展示 Gateway 状态摘要

### 5.3 Gateway 连接层

文件：

- `src/services/GatewayService.ts`
- `src/services/GatewayDeviceAuth.ts`
- `src/stores/gatewayStore.ts`

职责：

- 管理 WebSocket 连接生命周期
- 实现 OpenClaw 官方握手流程
- 处理 `connect.challenge` 和 `req/connect`
- 管理 shared token 与 paired-device token
- 心跳、断线重连、离线队列
- 统一分发 Gateway 响应和事件

当前连接方案的重要特点：

- 已不再使用旧的“连接后直接调用 `gateway.get_status`”方式
- 采用设备签名 + token 的官方连接模式
- 首次连接依赖 Settings 中的 `Gateway Token`
- 首次连接成功后会缓存 Gateway 返回的 paired-device token
- 遇到失效 device token 时会自动清理并重试

### 5.4 聊天模块

文件：

### 5.5 看板模块

### 5.6 文件模块

- 保存设置、错误日志、离线队列、设备身份、device token 等数据

## 6. 关键数据流说明

### 6.1 启动数据流

1. `App.tsx` 启动时调用 `gatewayStore.hydrate()`
2. store 调用 `gatewayService.hydratePersistence()`
3. 读取本地持久化数据
4. 执行 `connect()`
5. 连接成功后刷新 Agent 列表

### 6.2 Gateway 连接数据流

1. 建立 WebSocket
2. 等待服务端发送 `connect.challenge`
3. 本地生成或读取设备身份
4. 结合 token 与 challenge 构造官方 `connect` 请求
5. 成功后缓存服务端返回的 paired-device token
6. 启动心跳与后续业务请求

### 6.3 页面交互数据流

典型路径如下：

```text
组件点击 -> store action -> service 调用 -> Gateway / Persistence -> store 更新 -> 组件刷新
```

## 9. 维护说明

### 9.1 日常维护建议

- 修改页面展示，优先改 `components/`
- 修改状态组织，优先改 `stores/`
- 修改协议或业务逻辑，优先改 `services/`
- 修改环境切换逻辑，改 `src/config/runtime.ts`
- 修改部署方式，改 `deploy/linux/`、`README.md`、`README.zh-CN.md`、`SKILL.md`

### 9.2 修改 Gateway 协议时的注意事项

- 不要绕开 `GatewayService.ts` 直接在组件里写 WebSocket 逻辑
- 新增协议字段时，先补 `src/types/index.ts`
- 改握手逻辑时，必须同步检查：
  - `GatewayService.ts`
  - `GatewayDeviceAuth.ts`
  - `SettingsModule.tsx`
  - `GatewayService.test.ts`
- 协议改动完成后，至少执行：
  - `npm run test`
  - `npm run build`

### 9.3 修改设置项时的注意事项

- 先在 `src/types/index.ts` 的 `SettingsState` 中新增字段
- 再更新 `src/data/mock.ts` 默认值
- 然后更新 `src/stores/settingsStore.ts`
- 最后更新 `SettingsModule.tsx` UI

### 9.4 修改持久化逻辑时的注意事项

- 所有本地存储尽量统一通过 `PersistenceService.ts`
- 避免在多个 service 中直接操作 `localStorage`
- 新增持久化 key 时，命名尽量保持统一前缀

### 9.5 修改 Linux 部署说明时的注意事项

- `README.md`、`README.zh-CN.md` 与 `SKILL.md` 需要一起同步
- 如果改了反向代理路径，也要同步检查：
  - `deploy/linux/nginx/openclaw-center.conf`
  - `.env.example`
  - `src/config/runtime.ts`

## 10. 常见排障说明

### 10.1 无法连接 Gateway

优先检查：

- Gateway URL 是否正确
- 是否使用了正确的 `ws://` / `wss://`
- 远程环境是否处于 HTTPS 安全上下文
- `Gateway Token` 是否有效
- 服务端是否需要手动批准新设备

### 10.2 首次能连，后续不能连

优先怀疑：

- paired-device token 失效
- 服务端 token 策略变化
- 客户端本地缓存过旧

可以尝试：

- 重新填写 `Gateway Token`
- 清理本地缓存后重新连接
- 查看 Settings 中的错误日志

### 10.3 Studio 不显示

优先检查：

- `Studio URL` 是否可访问
- 是否允许 iframe 嵌入
- Nginx / CSP / `X-Frame-Options` 是否拦截

### 10.4 测试通过但线上不可用

优先检查：

- 线上是否使用 HTTPS/WSS
- 域名证书是否被客户端信任
- Linux 防火墙和安全组是否放通
- Nginx 是否正确转发 `/gateway/` 和 `/studio`
